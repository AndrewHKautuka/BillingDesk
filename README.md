# BillingDesk

A subscription billing management app for tracking plans, calculating renewal dates, and processing payments. The frontend is a single-page application (SPA) that communicates with a RESTful .NET backend API over HTTP.

## Tech Stack

**Backend**
- .NET 10, ASP.NET Core (Controller-based)
- PostgreSQL with Entity Framework Core (Npgsql + NodaTime)
- FluentValidation, Mapster, Bogus
- OneKhusa SDK (payment processing)
- Scalar / Swagger UI (API docs)

**Frontend**
- React 19 + React Router 7 (SPA - SSR disabled)
- TypeScript 6, Vite 8
- Tailwind CSS 4, shadcn/ui components
- TanStack Query, React Hook Form + Zod, Zustand, Axios

**Tooling**
- pnpm (frontend package manager)
- ESLint (flat config), Prettier, EditorConfig
- ReSharper / Rider (backend code style)
- JetBrains ReSharper CLI (`jb`) via .NET local tool

## Getting Started

### 1. Clone the repo

```bash
git clone https://github.com/AndrewHKautuka/BillingDesk [<directory>]
cd <directory>
```

> If `\<directory\>` was omitted in `git clone`, use `BillingDesk`

### 2. Set up the backend

Install the .NET local tools:

```bash
dotnet tool restore
```

Configure user secrets (see [Environment Variables](#environment-variables)):

```bash
cd BillingDesk-API/BillingDesk
dotnet user-secrets init   # if not already initialised
# Populate secrets - refer to secrets.example.json
```

Apply database migrations:

```bash
dotnet ef database update
```

Run the API:

```bash
dotnet run
```

The API will be available at `http://localhost:5169`. Interactive docs are at `/swagger/ui` (development only).

> Running the `https` run configuration is disrecommended

### 3. Set up the frontend

```bash
cd billing-desk
pnpm install
cp .env.example .env.development
# Edit .env.development with your values (see Environment Variables)
pnpm dev
```

The app will be available at `http://localhost:5173`.

## Linting & Formatting

### Frontend

**ESLint** uses a flat config (`eslint.config.js`) with strict TypeScript rules, React Hooks enforcement, `jsx-a11y` accessibility checks, TanStack Query rules, and Prettier compatibility.

**Prettier** (`.prettierrc`) - 80-character print width, no semicolons, double quotes, 2-space indent, LF line endings, trailing commas (ES5). Import order is enforced via `@trivago/prettier-plugin-sort-imports`, and Tailwind class sorting via `prettier-plugin-tailwindcss`.

```bash
pnpm lint:check      # check only
pnpm lint            # check and auto-fix
pnpm format:check    # check only
pnpm format          # check and auto-write
```

> Ignore the ESLint warning for `useReactTable` not being compatible with memoization. This is a known issue with the current implementation of TanStack Table and should not cause issues.

### Backend

Code style is enforced through **EditorConfig** (`.editorconfig` at the repo root) and **ReSharper / Rider** settings embedded in the same file.

- C# files: tabs, 4-space tab width, file-scoped namespaces, `var` preferred
- All other files: spaces, 2-space indent, LF, 120-char max line length
- ReSharper wraps arguments/parameters aggressively beyond 80 characters
- Migrations (and the DbContext Model Snapshot) are excluded from analysis (`generated_code = true`)

ReSharper CLI inspection (requires `dotnet tool restore`):

```bash
dotnet tool run jb inspectcode BillingDesk-API/BillingDesk/BillingDesk.csproj --output=report.xml
```

## Environment Variables

### Frontend - `.env.development`

Copy `.env.example` to `.env.development` and fill in the values. Vite only exposes variables prefixed with `VITE_`.

| Variable | Description |
|---|---|
| `VITE_API_URL` | Base URL of the backend API (e.g. `http://localhost:5169/api`) |
| `VITE_CONNECTORS` | Comma-separated list of payment connector `<6-digit-id>:<name>` pairs. At least one required. Example: `123456:My Connector` |

Example file: [`billing-desk/.env.example`](billing-desk/.env.example)

`.env*` files (except `.env.example`) are git-ignored.

> Environment variables are accessed via [billing-desk/app/config/env.ts](billing-desk/app/config/env.ts) which implements a fail-fast approach

### Backend - User Secrets & `appsettings.json`

Potentially sensitive configuration is kept out of source control using [.NET User Secrets](https://learn.microsoft.com/en-us/aspnet/core/security/app-secrets).

Initialise and populate secrets:

```bash
cd BillingDesk-API/BillingDesk
dotnet user-secrets set "ConnectionStrings:Primary" "User Id=postgres; Password=<password>; Server=localhost; Port=5432; Database=billing-desk;"
dotnet user-secrets set "FrontendUrl" "http://localhost:5173"
dotnet user-secrets set "OneKhusa:IsSandbox" "true"
dotnet user-secrets set "OneKhusa:ApiKey" "<your-api-key>"
# ... and so on. See secrets.example.json for all keys
```

Example file: [`BillingDesk-API/BillingDesk/secrets.example.json`](BillingDesk-API/BillingDesk/secrets.example.json)

> Direct configuration of the secrets.json file in development is recommended over using the CLI. For more information see [Microsoft Learn - Safe storage of app secrets in development in ASP.NET Core - Use the Secret Manager tool section](https://learn.microsoft.com/en-us/aspnet/core/security/app-secrets?view=aspnetcore-10.0#use-the-secret-manager-tool)

**`appsettings.json`** (committed, non-sensitive):

| Key | Description |
|---|---|
| `FxRates.RatesToMwk` | Exchange rates to MWK used by the FX rate provider (USD, EUR, etc.) |
| `Logging` | Log level configuration |

## Build & Deployment

The frontend and backend are deployed independently.

### Frontend

```bash
cd billing-desk
pnpm build
```

Output is written to `billing-desk/build/`. Because SSR is disabled, the build produces a fully static SPA that can be served from any static host (Nginx, S3 + CloudFront, Vercel, Netlify, etc.). Set `VITE_API_URL` to point at the deployed backend before building.

### Backend

```bash
cd BillingDesk-API/BillingDesk
dotnet publish -c Release -o ./publish
```

The published output in `./publish/` can be deployed to any host that supports .NET 10 (Linux/Windows server, Docker container, Azure App Service, etc.). Ensure the production environment provides the required connection string and secrets via environment variables or a secrets manager - do not commit `secrets.json`.

Database migrations should be applied before or during deployment:

```bash
dotnet ef database update
```

## Testing

No automated tests have been included in this project. Manual testing can be done via the Swagger UI at `/swagger/ui` or the Scalar API reference at `/scalar` (both available in development).

The backend automatically seeds a set of sample subscriptions on startup (~150) if the database contains none, so the app is usable out of the box without manual data entry.
